# pi-secure-it

Pi coding agent extension: two-layer security sandbox + skill docs. Package manifest is the source of truth for what's wired up — see `package.json` → `pi` field:

- `pi.extensions`: `security-guard.ts` (L2 in-process tool guard), `sandbox/index.ts` (L1 OS sandbox via sandbox-exec/bubblewrap)
- `pi.skills`: `skills/pi-secure-it/SKILL.md` (agent-facing how-to, auto-loaded by pi when relevant)

`extensions/staged/` = drafts/examples only, NOT referenced by `package.json` — do not treat as active code.

## Commands

```bash
npm run typecheck      # tsc against pi's bundled .d.ts
npm test               # security/tests/*.mjs (path-matcher, url-allowlist, symlink-escape)
./security/check.sh    # tests + regenerate SECURITY.md (or: `tests`|`render` only)
```

## Docs map — read the right one, don't duplicate across them

| File | Audience | Edit? |
|---|---|---|
| `SECURITY.md` | current-state dashboard (layer status, tests, gaps) | **No** — auto-generated by `security/render.mjs` from `security/manifest.json`. Edit the manifest, then run `./security/check.sh` |
| `skills/pi-secure-it/SKILL.md` | pi's own context when helping a user configure/debug the sandbox | Yes, hand-edit |
| `README.md` | human install/config reference | Yes, hand-edit |
| `adr/*.md` | why a design decision was made | Append new ADRs, don't rewrite old ones |

## Conventions

- Conventional Commits required (`commitlint`); `fix:`/`feat:`/`feat!:` drive semantic-release version bumps on `main`. `docs:`/`chore:`/`refactor:` don't release.
- New non-trivial design decisions → add an ADR in `adr/` (numbered, `Status: Accepted`), don't just leave it in a commit message.
- Policy file shape and layer semantics are documented once, in `skills/pi-secure-it/SKILL.md` — check there before re-explaining sandbox.json fields elsewhere.
